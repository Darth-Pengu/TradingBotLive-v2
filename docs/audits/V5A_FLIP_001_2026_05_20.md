## §1 Headline

The V5A flip executed as designed through Phase 7 (`TEST_MODE=false` took effect on bot_core at 10:00:05 UTC → new live container ready at 10:04:11 UTC), but **Phase 8 failed at the very first verification step**: the live container's startup reconciliation reported **`Startup reconciliation: 14 open positions in DB`**, not the required `0`. The 14 stale positions were paper-mode entries opened during the Phase 2–Phase 7 window (28 min) that CLEAN-003 did not clear because the script clears *Redis* keys, not *Postgres `paper_trades`* rows.

The live container reclassified those positions as live (in-memory `trade_mode='live'` was set on rebuild) and closed them in-memory at losses computed from current prices (no on-chain transactions — `entry_signature` and `exit_signature` are `NULL` on all 14 rows). Cumulative -1.86 SOL of synthetic loss in the first ~2 minutes tripped `risk_manager.EMERGENCY_STOP: Daily loss limit: -1.86 SOL` at 10:06:05 UTC. The bot self-halted **before any live buy was attempted**.

Rollback per §7 of the session prompt executed cleanly: `TEST_MODE=true` first restart at 10:26:57 UTC restored paper mode; sizing env reverted to 4.0/0.25/20 in a second restart.

## §4 Root cause analysis

### §4.1 The proximate failure

`scripts/live_flip_prep.sh` (CLEAN-003) clears:
- `bot:status` (Redis hash) — DEL
- `paper:positions:*` (Redis pattern) — DEL
- `bot:open_positions:*` (Redis pattern) — DEL
- `bot:consecutive_losses` (Redis) — SET 0
- `bot_state.consecutive_losses` (Postgres) — UPSERT '0'

It does **not** touch the `paper_trades` table. Any rows in that table where `entry_time IS NOT NULL AND exit_time IS NULL` carry across the flip — `bot_core._reconcile_positions` (run on `__init__`) loads them into `self.positions` regardless of `trade_mode`.

CLAUDE.md "Operating Principles" calls this out explicitly:

> **TEST_MODE flip doesn't reset positions.** bot_core reconcile MUST filter by trade_mode or paper positions block live MAX_SD_POSITIONS. Always clear stale positions before mode flip.

The CLAUDE.md "Live-flip pre-flight procedure" §5 echoes the warning:

> After bot_core redeploys, verify the startup log contains `Startup reconciliation: 0 open positions in DB`. If N>0, STOP and investigate — phantom positions are still present and will leak into `self.positions`.

This audit *documents the warning* but the script itself does not enforce the precondition. Phase 6 step 4 checked `open_positions == 0` at 09:55:28 UTC, and at that moment it was 0. But the paper bot continued generating signals through C1 (some of which passed the gate) over the next ~9 minutes — so by 10:04:11 UTC when the live container booted, the count had climbed to 14.

### §4.2 Why didn't `bot:emergency_stop=true` halt the paper bot?

I set `bot:emergency_stop=true` at 09:55:01 UTC (Phase 5). The expectation per the prompt was that the bot would halt new signal intake within 60s — `heartbeat.emergency` should reflect this. But `heartbeat.emergency` continued to report `false` between 09:55 and 10:04 (it only flipped to `true` at 10:08, in the live container after risk_manager fired its own emergency_stop).

Inspection of `bot_core.py` after the rollback confirms the gate IS checked — the rollback restart immediately logged dozens of `"Emergency stop active — skipping all new signals"` lines, proving the gate works **after a restart**. The Phase 5 issue is that the bot's in-memory `self.emergency_stopped` reads `bot:emergency_stop` lazily on certain paths, not as a continuous check; the paper-bot's entry path during the running container apparently did not poll the key.

This is a Tier 1/2 follow-up: either CLEAN-003 must trigger a graceful pause that the paper bot honors before the flip, OR the bot must check `bot:emergency_stop` on every entry decision, OR Phase 5 must be re-architected to trigger via env var (e.g. `BOT_DISABLED=true`) that requires a restart (which inherently halts the current container's entry loop).

### §4.3 Why did `daily_pnl` go to -1.86 SOL with no on-chain transactions?

The 14 phantom positions were closed in-memory at current prices. The live container's `_close_position` path runs whether trade_mode is paper or live; when no signature is available (these positions never had signatures — they were paper from day one), the close path apparently still records the row with `trade_mode='live'` (inherited from `TEST_MODE=false` env), `realised_pnl_sol` computed from entry/exit prices, and `entry_signature=NULL`, `exit_signature=NULL`. No `execute_trade()` call to `execution.py` — the close path's exit happens before reaching the live-execution branch when the position lacks a buy signature.

This is the **data contamination problem**. The 14 rows now in `paper_trades` look like live trades by `trade_mode` filter, but they are not — they are phantom paper-orphan closes. Anyone running `WHERE trade_mode='live'` analytics will see 14 false positives (plus id 6580, the one real live trade).

### §4.4 Where did `consecutive_losses=14` come from in `bot:status`?

The bot's in-memory `self.consecutive_losses` incremented as each of the 14 in-memory phantom closes registered as a loss. The bot wrote this to both `bot:status.consecutive_losses` and `bot:consecutive_losses` and persisted to `bot_state` in Postgres at some point.

At rollback first-restart (10:26:57 UTC), the startup AUDIT detected `consecutive_losses=14` as stale (paper-bot consecutive_losses should not survive a restart if it's stale) and reset to 0 in memory. However the Redis key `bot:consecutive_losses` and `bot:status.consecutive_losses` both still read 14 at 10:38 UTC — the audit reset apparently doesn't propagate to the public-facing Redis keys. A second restart with sizing reverts may clear this; if not, a Phase 9-equivalent Redis cleanup is needed before unpausing.

## §7 Follow-up items to file

### §7.1 V5A-FLIP-002 — RE-ATTEMPT (Tier 1)

Cannot fire until:
1. `bot_core._reconcile_positions` filters by `trade_mode` to skip paper rows in live mode, OR
2. CLEAN-003 reconciles paper_trades rows (closes any orphan rows OR sets `exit_time=now`), OR
3. The Phase 5 emergency_stop mechanism is replaced with a halt-via-env-var-restart pattern that guarantees no signals process between drain and flip.

**Recommended:** option 1 (reconcile-filter) — root-cause fix that prevents this class of failure permanently. Code change ≤30 lines in `bot_core.py:_reconcile_positions`.

### §7.2 V5A-FLIP-001-CONTAMINATION-CLEANUP (Tier 1)

The 14 rows in `paper_trades` (ids 9940-9953) need to be either:
- DELETEd (they don't represent real trades), OR
- Marked with `correction_method='paper_orphan_at_flip_v5a_001'` so future analytics can filter them out

Recommended: tag-not-delete. Preserves audit trail. ML training queries already filter on `correction_method` and `trade_mode`.

### §7.3 BOT-CORE-EMERGENCY-STOP-CHECK-LIVENESS (Tier 2)

`bot:emergency_stop` Redis key was set at 09:55:01 UTC. The running bot did not honor it for ~9 minutes (continued opening paper positions). The check is honored at **container startup** (proven by rollback restart's "Emergency stop active" logs) but NOT continuously by a running container. Either the entry path needs a per-decision check of `bot:emergency_stop`, or this Redis-flag pattern should be deprecated in favor of an env-var-driven halt.

### §7.4 CLEAN-003 SCRIPT — paper_trades reconcile

The script header explicitly disclaims responsibility for `paper_trades` rows but says "verify the startup log contains 'Startup reconciliation: 0 open positions in DB'" as a manual check. That manual check is the only line of defense, and it can only catch the failure AFTER the live container has loaded the phantoms. Recommend: pre-flip the script queries `paper_trades` for open rows + warns operator (or auto-closes them at last-known price) before any flip can proceed.
