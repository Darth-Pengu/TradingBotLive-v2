## §2 Phase 1 — ORPHAN-PAPER-CLOSURE-INVESTIGATION findings

Full detail: `docs/audits/ORPHAN_PAPER_CLOSURE_INVESTIGATION_001_2026_05_21.md`.

Four-question summary:

**Q1 — Reconciler exact query.** The reconciler (and `_load_state`) reads `paper_trades` when paper-mode, `trades` when live-mode (line 246 of `bot_core.py`). The pre-fix `mode_clause` (line 249, `_load_state` line 308) only applied trade_mode filter when reading `paper_trades` — not when reading `trades`. Per-V5A-FLIP-001-V2 commit `4b647a7` (2026-04-16) added the partial filter; it was incomplete.

**Q2 — Pre-incident state of 14 orphans.** The 14 paper_trades rows ids 9940-9953 are **NEW INSERTs by the live container's defensive close path**, not pre-existing rows that had `trade_mode` flipped. Original paper_trades rows for those mints existed earlier with lower IDs (correctly closed in paper mode). The 14 contaminated rows are a parallel record of the live container's in-memory close events at 10:04-10:05 UTC on 2026-05-20. The 14 corresponding `trades` rows (t_ids 6654, 6885, 7497, 7500, 7762, 7764, 7782, 8422, 9414, 9415, 9926, 9927, 9929, 9930) were the original paper trades that remained "open" in the `trades` corpus (closed_at=NULL) due to Q3.

**Q3 — Paper close path conditional.** `paper_sell:433-443` always updates `paper_trades.exit_time`. `bot_core._close_position:1247-1257` updates `trades.closed_at` **only if `pos.trades_ml_id` is truthy**. When `trades_ml_id == 0` (default value, set when position lacks an ML record ID), the `trades` row stays "open" forever. This is the accumulation mechanism for `trades`-table paper orphans. Data hygiene issue, not structural design. Filed Tier 3 follow-up `PAPER-CLOSE-TRADES-CLOSED_AT-HARDENING-001`.

**Q4 — Current state via reconciler criteria.**
- `paper_trades WHERE exit_time IS NULL AND trade_mode='paper'`: 0 rows (paper-mode reconciler clean).
- `trades WHERE closed_at IS NULL` (pre-fix live-mode reconciler): 1 row at investigation time (different mint by verify-time — proving accumulation is ongoing).
- `trades WHERE closed_at IS NULL AND trade_mode='live'` (post-fix live-mode reconciler): 0 rows.

The ongoing 1-row trades-table orphan is exactly the situation the Bug 1 fix is designed to handle. Deploying the fix in paper mode is safe (paper-mode SQL behavior unchanged) and prevents future live-mode contamination.

**STOP evaluation:** INV1/INV2/INV3 all evaluated, none fired. Proceeded to Phase 2 + Phase 3.

## §6 Phase 5 — Bug 3 outcome

**Bug 3 does not exist.** Investigation found the per-decision `bot:emergency_stop` Redis check is already implemented at `services/bot_core.py:604-609` inside `process_signal`. The V5A-FLIP-001-V2 audit's "Phase 5 finding (root cause #2)" was based on two misinterpretations:

1. Heartbeat reporting `emergency: false` was conflated with the per-decision check status. Heartbeat reflects in-memory `self.emergency_stopped`, not the Redis flag. The Redis-flag check IS honored at signal-entry time (proven by post-rollback logs at 10:26:57Z showing "Emergency stop active — skipping all new signals").
2. The 14 phantom positions had `entry_time` from 2026-05-12 to 2026-05-19 — DAYS before V5A-FLIP-001-V2. They were NOT opened during the 9-minute drain window. The per-decision check on Phase 5 worked correctly; the phantoms came from pre-existing `trades`-table orphans (Q4 / Phase 1 finding), not from drain-window position-opening.

STOP-Scope5 fires for the right reason: the fix would change zero lines because the behavior is already correct.

Filed observability follow-up `HEARTBEAT-EMERGENCY-STOP-REFLECTION-001` Tier 3 — make `heartbeat.emergency` surface the Redis flag too, so operators don't get confused. Not V5A-blocking; deferred.
