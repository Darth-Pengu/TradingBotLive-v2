## Root cause — `LIVE-DRAWDOWN-PEAK-SEED-001` (live-only; NOT paper-observable)
`_load_state` (`bot_core.py`): the snapshot read (`ORDER BY id DESC LIMIT 1`, **no trade_mode filter**) loads the latest **paper** snapshot (~132 SOL) and sets `peak_balance_sol` to it. The #12 live block then correctly seeds `total_balance_sol` from on-chain (5.064) — **but set `peak_balance_sol = max(peak, _onchain)`, which KEPT the stale 132 peak.** `risk_manager._drawdown_multiplier` then computed `(132 − 5.064)/132 ≈ 96.4%` → > 20% → `EMERGENCY_STOP: STOP ALL TRADING`.

Paper never exposes this: paper resets `peak_balance_sol = total_balance_sol` on every startup (lines 321-323, 375-377) precisely to avoid accumulated-loss false stops. The live path skipped that reset → the supervised first-flip is the only place it could surface. **This is exactly the class of flip-confirmed-only defect the supervised first-flip exists to catch.**

## Fix (applied — `bot_core.py` live `_load_state` block, paper-safe)
On live boot, **reset** `peak_balance_sol = _onchain` (the on-chain seed) instead of `max(peak, _onchain)`. On-chain is the live reality; the stored peak is paper-contaminated (unfiltered snapshot). The durable cross-restart kill-switch is `DAILY_LOSS_LIMIT_SOL` (#11, reloaded from today's live closes), so resetting the peak does not weaken the hard loss cap. py_compile + structural verified. Live-only (`if not TEST_MODE:` block) → paper byte-for-byte unchanged.
